Étape 9 : Relations

Objectifs

  • gérer les relations avec L'ORM

Quel est le problème ?

Jusqu'ici, nous n'avons pas vraiment géré de story qui fasse appel aux relations entre entités.
Cette problématique est pourtant courante : nos entités sont rarement "intdépendantes". Par exemple, il y a une relation évidente entre ProgrammeDeCinemas et Cinema : un élément de programme s'applique à un cinéma.
De même entre ProgrammeDeCinemas et Film
Il nous faut transcrire dans le code ces relations, et il nous faut nous assurer de leur stockage dans notre couche de persistance : c'est là qu'interviennent les relations entre les tables de notre base de données, qui vont être abstraites par l'ORM.

Cependant, nous allons traiter cette relation par un concept métier et pas par une contrainte technique liée au fait que l'on stocke des données dans une base.

Retour sur la story 3

En tant que responsable de la programmation, je veux pouvoir connaître la liste des films à l’affiche de ce cinéma
Pour un cinéma, en dessous de ses informations détaillées on veut un bloc avec la liste des films à l’affiche

Nous allons afficher la programmation d'un cinéma. Pour cela, il nous faut la liste de ses films à l'affiche.
Pour l'instant, nous disposons d'une interface de ProgrammeDeCinema, mais le concept de film à l'affiche n'est pas encore présent.

Si l'on se place du point de vue de la base de données, on a une relation many to many entre les cinémas et les films : un film peut être à l'affiche de plusieurs cinémas, et un cinéma peut avoir plusieurs films à l'affiche.

Si l'on décortique ce que ça signifie, on voit que la relation entre film et cinéma "est" notre concept de "film à l'affiche" :

Ou alors, vu autrement, si l'on représente des instances de films à l'affiche :

Dès lors, on peut considérer notre ProgrammeDeCinema comme un repository de films à l'affiche.

Un test fonctionnel

Il nous faut un test pour avancer. Ecrivons un test fonctionnel.
On va commencer par tester que la page de détail d'un cinema est disponible :

public function test_page_cinema_est_disponible()
{
    $client = static::createClient();
    $idCinema=$this->unCinema()->getId();
    $client->request('GET', '/admin/cinemas/'.$idCinema);
    $this->assertEquals(200, $client->getResponse()->getStatusCode());
}

On utilise dans ce test une méthode unCinema
Mais elle sort d'où ?
En fait, nous avons besoin d'un cinéma pour pouvoir utiliser son identifiant (que vous avez ajouté à l'étape 7) dans la route qui affiche le détail d'un élément.

Nous avons donc ajouté un méthode qui fait appel aux outils fournis par les classes de tests de Symfony pour aller chercher le premier cinéma disponible :

private function unCinema():Cinema
{
    $container = self::$container;
    /** @var AnnuaireDeCinemas $annuaire */
    $annuaire=$container->get("App\Domain\AnnuaireDeCinemas");
    return $annuaire->tousLesCinemas()[0];
}

...Pour que cette méthode fonctionne, il nous faut évidemment des fixtures, sinon, pas de Cinema disponible.

Créez une nouvelle méthode dans notre controleur pour que ce test passe au vert, avec le template associé.

Elle doit ressembler à ça :

/**
 * @Route("/admin/cinemas/{cinema}", name="detail_cinema")
 */
public function detailCinema(Cinema $cinema){

    return $this->render('Cinema/cinema.html.twig',[
        'cinema'=>$cinema,
    ]);
}

Au passage, notez que Symfony sait instancier un cinéma complet si on lui passe le bon type dans la méthode, alors que ce qu'il reçoit n'est que l'identifiant.

Maintenant, nous pouvons nous préoccuper de savoir si les films à l'affiche sont effectivement affichés sur la page de détail du cinéma :

public function test_la_page_d’un_cinema_affiche_les_films_a_l’affiche()
{
    $idCinema=$this->unCinema->getId();
    $titreFilm=$this->unFilm->getTitre();

    $crawler=$this->client->request('GET', '/admin/cinemas/'.$idCinema);

    // Il faut vérifier que la contenu de la page contient le titre du film.
    $this->assertGreaterThan(
        0,
        $crawler->filter('html:contains("'.$titreFilm.'")')->count()
    );
}

Il y a eu un peu de refactoring ici : on n'utilise plus dans ce test une méthode pour avoir le cinéma, mais une propriété privée.
Mais elle sort d'où ?
...de la méthode setUp de notre classe de test.

La methode setUp est une méthode "spéciale" : PHPUnit l'exécute avant chaque test : elle nous permet donc de remplir des préconditions qui sont communes à plusieurs tests dans la même classe de test.
Elle évite donc la duplication de code.

Examiner la classe dans son ensemble permet de mieux comprendre :

<?php
namespace App\Tests\Controller;
use App\Entity\Cinema;
use App\Entity\Film;
use Symfony\Bundle\FrameworkBundle\Test\WebTestCase;

class CinemaAdminControllerTest extends WebTestCase
{
    /** @var Cinema */
    private $unCinema;
    /** @var Film */
    private $unFilm;
    private $client;

    protected function setUp(): void
    {
        $this->client = static::createClient();

        $container = self::$container;
        $this->unCinema = $container->get("App\Domain\AnnuaireDeCinemas")->tousLesCinemas()[0];
        $this->unFilm = $container->get("App\Domain\CatalogueDeFilms")->tousLesFilms()[0];
    }

    public function test_page_cinemas_est_disponible()
    {
        $this->client->request('GET', '/admin/cinemas/');
        $this->assertEquals(200, $this->client->getResponse()->getStatusCode());
    }
    public function test_page_cinema_est_disponible()
    {
        $idCinema=$this->unCinema->getId();
        $this->client->request('GET', '/admin/cinemas/'.$idCinema);
        $this->assertEquals(200, $this->client->getResponse()->getStatusCode());
    }
    public function test_la_liste_des_cinemas_affiche_les_cinemas()
    {
        $crawler=$this->client->request('GET', '/admin/cinemas/');

        // Il faut vérifier que la contenu de la page contient notre cinéma.
        $this->assertGreaterThan(
            0,
            $crawler->filter('html:contains("MegaCGR")')->count()
        );
    }
    public function test_la_page_d’un_cinema_affiche_les_films_a_l’affiche()
    {
        $idCinema=$this->unCinema->getId();
        $titreFilm=$this->unFilm->getTitre();

        $crawler=$this->client->request('GET', '/admin/cinemas/'.$idCinema);

        // Il faut vérifier que la contenu de la page contient le titre du film.
        $this->assertGreaterThan(
            0,
            $crawler->filter('html:contains("'.$titreFilm.'")')->count()
        );
    }
}

Grâce à la méthode setUp, les Film et Cinema sont donc disponibles pour chaque test, de même que le client.

Ce test échoue : il nous faut afficher les films dans la page de détail d'un cinéma.

Donc dans notre controleur, il nous faut obtenir la liste des films à l'affiche de notre cinéma.

...Donc il faut injecter notre ProgrammationCinemaHandler dans la méthode :

/**
 * @Route("/admin/cinemas/{cinema}", name="detail_cinema")
 */
public function detailCinema(
    Cinema $cinema,
    ProgrammationCinemaHandler $programmationCinemaHandler
){
    
    $programmeQuery=new ProgrammationCinemaQuery($cinema);
    $filmsAAffiche=$programmationCinemaHandler->handle($programmeQuery);
    
    return $this->render('Cinema/cinema.html.twig',[
        'cinema'=>$cinema,
        'filmsAAffiche'=>$filmsAAffiche,
    ]);
}

On est est en erreur 500.

On regarde dans le navigateur pour savoir pourquoi, et Symfony nous explique que nous n'avons pas d'implémentation de App\Domain\ProgrammeDeCinema

Implémentons un ProgrammeDeCinemas

Nous avons vu en introduction que notre programme de cinéma est en fait un repository de films à l'affiche, et qu'un film à l'affiche est un lien entre un cinéma et un film.

Nous allons donc créer cette entité "FilmAAffiche".

Pour accélerer le processus, nous allons utiliser le générateur de code fourni par le framework.

bin/console make:entity

Ce générateur nous demande les renseignements nécessaires pour créer notre entité. Elle comporte deux champs, qui sont tous les deux de type "relation" (te type ManyToOne) : cinema et film

Nous avons créé une nouvelle entité gérée par Doctrine, il nous faut mettre à jour le schéma de base :

bin/console doc:sch:upd --force

Le générateur de code crée aussi un repository, qu'il nomme par défaut FilmAAfficheRepository
On le renomme en DoctrineProgrammeDeCinemas pour plus de cohérence (attention à la référence à ce nom dans les annotation de la classe FilmAAffiche)

ce DoctrineProgrammeDeCinemas doit implémenter notre interface ProgrammeDeCinemas :
on demande à notre IDE de lui fournir des implémentations "basiques" de méthodes nécessaires (ou on les écrit à la main si on a pas un IDE digne de ce nom...) :

<?php

namespace App\Repository;

use App\Domain\ProgrammeDeCinema;
use App\Entity\Cinema;

use App\Entity\FilmAAffiche;
use Doctrine\Bundle\DoctrineBundle\Repository\ServiceEntityRepository;
use Doctrine\Common\Persistence\ManagerRegistry;

class DoctrineProgrammeDeCinema extends ServiceEntityRepository implements ProgrammeDeCinema
{
    public function __construct(ManagerRegistry $registry)
    {
        parent::__construct($registry, FilmAAffiche::class);
    }

    public function getFilmsPourCinema(Cinema $cinema): iterable

        return $this->findBy([
