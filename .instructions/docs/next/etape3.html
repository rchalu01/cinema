Étape 3 : Patterns et principes solid

Objectifs

  • comprendre et appliquer les principes SOLID ;
  • se documenter sur les design patterns ;
  • appliquer quelques design patterns.

Ce que vous devez livrer

  • Des modèles sous forme de schémas (ce que vous voulez qui exprime clairement votre pensée)
  • Un peu de code

L'important est de vous poser des questions et de comprendre les notions : n'hésitez pas à échanger entre vous, et à rechercher sur le web des ressources sur les notions qui sont obscures.

Les design patterns ne sont pas des solutions miraculeuses : veillez à les reconnaitre et à les appliquer quand ils sont nécessaires, lors de vos phases de refactoring.

SRP

Nous allons dans cette séance réfléchir à des améliorations de conception respectant les principes SOLID.

Soit la modélisation de conception suivante d’une gestion d’utilisateur :

User
+login(username,password)

Avec comme logique pour la méthode login :

  • Si la session n'existe pas encore il faut l'initialiser.
  • Rechercher dans la table user un utilisateur avec ce couple username/pass
  • Si il y a des résultats, enregistrer utilisateur courant dans la session sinon renvoyer une erreur.

Pattern singleton

Dans une application, on souhaite utiliser une connexion persistante à une base de données, car les coût (en temps) de connexion sont importants. La connexion est encapsulée dns une classe DBConnexion

Dans ce cadre, il est nécessaire que chaque module de l'application ayant besoin d'interagir avec la base de donnée utilise toujours la même instance de cette classe : ceci permet de n'ouvrir la connexion qu'une seule fois, même si plusieurs modules indépendants font usage de DBConnexion.

Proposez une modélisation de l’application du pattern singleton à notre problème de connexion.

Solid et patterns DI et factory

Nous souhaitons maintenant vendre des véhicules

Notre système de vente de véhicules gère des véhicules fonctionnant à l’essence, au diesel et des véhicules fonctionnant à l’électricité. Cette gestion est pour l’instant confiée à classe Car.

<?php

class Car
{
    public function __construct($engineType) {
        switch ($engineType){
            case "fuel":
                $this->engine=new FuelEngine();
                break;
            case "diesel":
                $this->engine=new DieselEngine();
                break;
            case "electric":
                $this->engine=new ElectricEngine();
                break;
        }
    }
}

Principe de ségrégation des interfaces (ISP)

Les client ne devraient pas dépendre de méthodes qu’ils n’utilisent pas. On pourrait presque y voir une forme d’héritage fonctionnel: une interface ne devrait pas déclarer plus d’un ensemble cohérent de méthodes. On parle aussi d’interfaces de rôles.

<?php

interface UserInterface
{
    public function login($user, $password);
    public function logout();
    public function isConnected();
    public function isAdmin();
    public function getRights();
}

class User implements UserInterface
{
    ...
}

Queries, Commands et CQS

Le paradigme objet repose sur la transcription dans le code des objets ou concepts de la vie "réelle" : utilisateur, moteur, ou même connexion à une base de donnée comme ci-dessus.
Les patterns query (requête) et command (commande) proposent de modéliser des "demandes" en en faisant des objets.
Vous pouvez les considérer comme la matérialisation sous forme d'objet (dans le code) d'une intention.

Par exemple, une commande peut être enregistrerUnPaiement dans une boutique en ligne ou ajouterUnePage dans un logiciel de traitement de texte.
Des exemples de queries : listerLesDesserts sur un site de restaurant, ou soldeDuCompte sur une interface bancaire.

La distinction query et command concerne les effets induits par la prise en charge de cette demande par le système :

  • une query renvoie une réponse, mais est sans effet sur l'état du système
  • une command induit un changement de l'état global du système, et n'a pas vocation à renvoyer de réponse

Le pattern CQS est complémentaire, et concerne à la base les méthodes d'un objet : on considère qu'une interaction avec un objet (donc l'appel à l'une de ses méthodes) est SOIT une command, SOIT une query, mais ne peut pas être les deux.

Étendre ce principe plus globalement à toute interaction avec le système, en modélisant les commands et queries sous forme d'ojets permet de modéliser du code plus facilement testable, et donc plus maintenable.
